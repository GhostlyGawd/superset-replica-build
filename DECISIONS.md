## ADR-0001 — Build lives in its own repo, not a worktree of the harness
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Spec §3.2 says create an isolated worktree so the orchestrator's checkout stays clean. But the orchestrator's repo here is `recursive-harness` (a meta/harness repo), not the product. A git worktree would bind the product's history to the harness.
- **Decision:** Create the build as its **own fresh git repo** in a sibling dir `D:/GitHub Projects/superset-replica-build`. This satisfies the intent (harness checkout stays clean) without polluting either repo's history.
- **Consequences:** The product gets its own GitHub repo and history (spec §3.3 wants this anyway). The harness repo is untouched.

## ADR-0002 — Windows-first cross-platform is in scope (beyond original)
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Original Superset is macOS-only Electron. Spec §0.8 mandates native Windows 10/11 + macOS + Linux, proven by CI + Windows e2e.
- **Decision:** Engine + desktop client + mobile-control workflow all target Windows/macOS/Linux from day one. No POSIX-only assumptions. CI matrix includes `windows-latest`, `macos-latest`, `ubuntu-latest` from Phase 0.
- **Consequences:** Forces node-pty (ConPTY), tree-kill, path-API discipline, EOL normalization, Windows-shell terminal support, and Node/Bun task scripts instead of `.sh` on user paths.

## ADR-0003 — Docker unavailable → PGlite as the self-hosted Postgres
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Spec §3.6/§5 want self-hosted Postgres + sync via Docker (no paid cloud DB). Preflight: Docker is **not installed** and the daemon is not running on this Windows host. Docker Desktop install needs admin + WSL2 + reboot — not achievable non-interactively, and a heavy bootstrap dependency that fights the "stand it up without touching your PC" goal.
- **Decision:** Use **PGlite** (Apache-2.0, Postgres compiled to WASM, runs in-process in Node/Bun) as the self-hosted Postgres for local/dev and single-host deployments. Access via **Drizzle ORM** (Postgres dialect) so the exact same schema/queries also run against a real Postgres server when one is present (e.g. remote host). This is a documented OSS substitution, not a downgrade: PGlite *is* Postgres, self-hosted, zero external services, zero paid cloud.
- **Consequences:** One-command bootstrap needs no Docker. `DATABASE_URL` selects PGlite (default, `file://./.data/pg`) or a real Postgres. Migrations run on both via Drizzle Kit. Docker compose remains available as an optional deployment path (documented) but is not required.

## ADR-0004 — Runtime: Bun primary, Node 24 fallback; no `.sh` on user paths
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Spec stack guidance = Bun + Turborepo. Bun installed via npm during preflight; Node 24 also present.
- **Decision:** Bun for install/test/scripts and the engine where it helps; all task scripts authored in TypeScript (run via Bun/Node) so they work on PowerShell/cmd and POSIX shells alike. No bash-only scripts on any user-facing path (spec §1).
- **Consequences:** Bootstrap and presets are cross-platform by construction.

## ADR-0005 — Desktop shell: Electron (Tauri rejected)
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Need a cross-platform desktop client with a Windows installer. Tauri requires the Rust toolchain (absent; heavy non-interactive install). Electron is Node-based and already aligned with the stack.
- **Decision:** Electron + `electron-builder` producing an **NSIS** Windows installer plus macOS (dmg/zip) and Linux (AppImage/deb) artifacts. Code-signing optional (no paid certs); document SmartScreen/Gatekeeper implications.

## ADR-0006 — Mobile: PWA-first, Capacitor optional

## ADR-0007 — PTY + process-tree kill (cross-platform)
- **Date:** 2026-06-14 · **Status:** Accepted (with RISK flag)
- **Context:** Engine must spawn/supervise CLI agents via PTYs and kill process trees on every OS.
- **Decision:** `node-pty` (ConPTY on Windows) for PTYs; `tree-kill` (uses `taskkill /T /F` on Windows, signal-tree on POSIX) for termination.
- **RISK:** `node-pty` native prebuilds may not exist for Node 24. **Phase 2 opens with a validation gate**: confirm node-pty loads on this Windows host; if not, fall back to `@homebridge/node-pty-prebuilt-multiarch` or pin the engine to Node 22 LTS (documented).
- **Consequences:** Process supervision is correct on Windows (no SIGTERM tree semantics assumed).

## ADR-0007a — PTY validation gate outcome: package + runtime pinned
- **Date:** 2026-06-14 · **Status:** Accepted (resolves the ADR-0007 RISK flag)
- **Context:** Phase 2 opened with the ADR-0007 load-validation gate (Windows, Node 24). Findings: (1) official `node-pty@1.1.0` is N-API and its **bundled** win/mac prebuilds load cleanly on Node 24 — the feared "no Node-24 prebuild" does **not** occur — but it ships **no Linux prebuild** (compiles via node-gyp). (2) **Decisive blocker not in ADR-0007:** node-pty **cannot run under Bun on Windows** — its ConPTY data pipe rides a `net.Socket` that Bun tears down (`ERR_SOCKET_CLOSED`); reproduced identically on every node-pty fork. See `evidence/phase-2/pty-validation.md`.
- **Decision:** Use **`@homebridge/node-pty-prebuilt-multiarch`** (ships prebuilds for win32-x64 + darwin + linux-x64/musl incl. the Node 24 ABI → no compiler on any CI runner) with **`tree-kill`**. Run the `pty-supervisor` **under Node 24, never Bun** (it already lives in a crashable child process per architecture §1/§5). Add the package to Bun **`trustedDependencies`** so win/mac prebuilds download at install (Linux is offline-bundled); add `actions/setup-node@v4` (node 24) to CI so the PTY integration test drives shells through Node.
- **Consequences:** PTYs spawn/stream/resize/tree-kill correctly on Windows/macOS/Linux with no build toolchain in CI. Monorepo stays on Bun for install/build/most tests; only the PTY-touching process is pinned to Node. Watch items: `prebuild-install` is deprecated upstream (still functional; matrix actively maintained), and win/mac prebuilds are a network download at install. Re-evaluate official `node-pty` if/when it bundles a Linux prebuild.

## ADR-0010 — Product name: Grove (codename SWARM retired)
- **Date:** 2026-06-14 · **Status:** Accepted
- **Context:** Phase 1 must choose the final, shippable product name (spec §6.3 / brief). It has to be original, ownable, easy as a CLI verb, and must not evoke "Superset." The codename **SWARM** is on-message but weakly ownable — it collides with Docker Swarm and Foursquare Swarm and is a generic noun.
- **Decision:** Ship as **Grove** (CLI: `grove`). A git **worktree** is literally a *tree*; many worktrees branch off one repository and share a single `.git` object store — one root system, many trees, i.e. a grove. The product is where a swarm of agents works that grove. The name is original, ownable, a clean CLI verb, does not evoke Superset, and — unlike a bolted-on label — anchors a coherent design thesis ("mission control for a swarm; calm surface, swarming depth"). "Swarm" is retained as common-noun product vocabulary, not the brand.
- **Consequences:** `STATE.json.project.final_name = "Grove"`. Brand + identity recorded in `docs/brand/` (name, story, voice, logo + hand-authored SVG mark/wordmark); the full visual system in `docs/design-system.md`; implementation in `packages/ui` (`@swarm/ui` v0.2.0) with the `apps/showcase` proof. The `superset-replica` repo name and Linear project keep their identifiers; only the user-facing product name changes.

## ADR-0011 — Engine conventions + CI-gate lessons (Phase 2)
- **Date:** 2026-06-15 · **Status:** Accepted
- **Context:** Phase 2 (host engine) surfaced three cross-cutting issues during integration that are worth pinning so later phases don't repeat them.
- **Decisions:**
  1. **Explicit `.ts` import extensions everywhere + `allowImportingTsExtensions: true` in `tsconfig.base.json`.** The host engine runs the PTY layer under Node (ADR-0007a), and Node's strip-types execution of worker scripts requires resolvable relative specifiers (explicit extensions). Mixing extensionless library source with `.ts`-extension workers caused TS5097 in consumer packages. Standardizing on explicit `.ts` everywhere is the only convention that works uniformly across bun (tests/bundling), Node (PTY workers), and `tsc` (`noEmit`, since builds use `bun build`, not tsc emit).
  2. **CI-gate discipline: never trust a cached local run.** Local `bun run <task>` goes through the Turbo cache and can return FALSE-GREEN cache hits that hide failures CI catches cold. The gate before any commit is a cache-disabled run: `bun run lint` (root Biome — not a Turbo task) + `bunx turbo run typecheck build test --force`, then confirm the 3-OS CI run after pushing. Per-package green ≠ full-tree green; local green ≠ CI green.
  3. **Windows file-lock resilience: git invocations use bounded transient-only retries.** Under heavy parallel test load (many concurrent PTY/process-spawning suites), git commands (`add`/`commit`/worktree ops) flake on Windows from `index.lock` / in-use-handle / AV contention. The git-worktree engine + fixtures retry only transient errors (errno/stderr signature allowlist) up to 5× with backoff; deterministic errors surface immediately. Real agents on Windows hit the same locks, so this lives in the engine, not just tests.
- **Consequences:** All packages build/run cross-platform under Node + Bun; the commit gate is reliable; the engine is robust to Windows filesystem contention. Recorded after the Phase-2 host engine reached a green 3-OS CI with the parallel-agents integration test (P01/P02/P04/P10/P11) passing.

## ADR-0012 — Phase-3 red CI is a GitHub Actions BILLING block, not a code defect; clean-install repro is the interim gate
- **Date:** 2026-06-15 · **Status:** Accepted (with an EXTERNAL BLOCKER that needs user action)
- **Context:** `main @ 0575214` (Phase-3 desktop foundation + terminal P05/diff P06) showed RED CI on all 3 OSes, and the prior session hypothesised a *clean-install-only code defect* (it could not retrieve CI logs — `gh BlobNotFound`). Re-derivation this session: both failed runs (`27539598023`, `27541523156`) carry the **identical annotation** on all three jobs — *"The job was not started because recent account payments have failed or your spending limit needs to be increased."* The jobs **never started** (2–3 s each → no logs, hence the BlobNotFound). The last actually-executed run was v0.3.0 (`27537264984`, 09:35Z); billing was exhausted before the two desktop pushes. The repo is **private**, so GitHub-hosted Actions minutes (esp. Windows/macOS, billed at 2×/10×) are metered and were cut off.
- **Decision:**
  1. **Root cause = account billing, NOT code.** Proven by a full clean-install repro mirroring CI on this Windows host (`rm -rf` all `node_modules` + `.turbo` + `*.tsbuildinfo`, `bun install --frozen-lockfile`, then `bun run lint` → `typecheck` → `build` → `test`): **ALL GREEN cold** — install ✓, lint ✓ (166 files), typecheck ✓ (17/17, 0 cached), build ✓ (17/17, 0 cached), test ✓ (9/9 task groups; all real PTY/host/worktree/diff integration tests pass). The prior clean-install-defect hypothesis is **refuted**. Prediction `e9ae9e0e` = hit.
  2. **The clean-install repro is the interim gate** while CI is billing-blocked: a cold `rm node_modules+.turbo+tsbuildinfo` + `bun install --frozen-lockfile` + the four CI steps is exactly what CI runs, so a green local cold run is the strongest available proxy. Build work continues against it. **But** macOS/Linux cannot be locally verified, so **no version may be cut and no Linear item marked Done on local-only evidence** — the non-negotiable "green windows+macos+ubuntu CI" stands; those milestones are *staged*, not *closed*, until billing is restored and CI runs green.
  3. **Remediation requires the user (external account state — cannot be fixed in-repo, autonomously, or via `gh`).** Two paths, surfaced to the user: **(a)** raise the Actions spending limit / fix the payment method at github.com/settings/billing; or **(b)** make the repo **public** (free unlimited GitHub-hosted Actions incl. Windows+macOS), which also aligns with the OSS-replica goal — but publishing the source is an outward-facing decision reserved to the user, **not taken autonomously**.
- **Consequences:** Phase 3 (and every later release gate) is **blocked on billing**, not on engineering. The orchestrator continues building + locally-verifying staged work (wave B2, e2e CI job, QA) so that the moment billing is restored, one push validates 3-OS-green and v0.4.0 cuts immediately. STATE.json `blocked` + RESUME.md PRIORITY updated to stop any future re-misdiagnosis.

## ADR-0013 — Desktop wave B2: open-in-external runs on the HOST; single hotkey registry; PGlite-persisted shortcuts (P08/P09)
- **Date:** 2026-06-15 · **Status:** Accepted
- **Context:** Phase-3 wave B2 adds workspace navigation + open-in-external (P08) and customizable keyboard shortcuts + a settings surface (P09) to the desktop renderer, wired to the REAL host (no mocks on user paths). Several design points were decided autonomously per the wave brief.
- **Decisions:**
  1. **Open-in-external executes on the HOST, not the Electron main/renderer.** A new host procedure `workspaces.openExternal({ workspaceId, target: "editor"|"terminal"|"folder" })` opens the worktree via `node:child_process` (NOT a PTY): editor = `$VISUAL`/`$EDITOR` else `code`/`cursor`; terminal = Windows Terminal/`cmd`, macOS `Terminal.app`, Linux `$TERMINAL`/`x-terminal-emulator`/common emulators; folder = `explorer`/`open`/`xdg-open`. **Rationale:** the worktree physically lives on the host, so it must open there — this is correct for both a local host and a future remote host, whereas opening from the Electron main process would only work locally. Binaries are resolved defensively (`resolveExecutable` → `where.exe`/`which`, PATHEXT-aware); launches are detached + unref'd; Windows `.cmd`/`.bat` shims are routed through `cmd.exe`.
  2. **Input field named `workspaceId` (not `id`).** The brief sketched `{ id, target }`; chose `workspaceId` for consistency with every other live-router procedure (`diffs`/`terminal`/`sessions` all key on `workspaceId`) and with the `@swarm/api` contract. Target enum is exactly `editor|terminal|folder`; the `@swarm/api` `ExternalTarget` was aligned from the stale `vscode|cursor|...`.
  3. **`projects.open` validates a REAL git repo, then seeds a worktree.** `projects.open({ path, name? })` runs `git rev-parse --is-inside-work-tree` / `--show-toplevel` / `--abbrev-ref HEAD` (no fake validation), registers the repo as a project idempotently by resolved root, and cuts a first isolated worktree from its current branch via the orchestrator. Repo path is entered as text (a native Electron folder picker is deferred to Phase 5; text entry is portable and works in the browser-served e2e). New / Quick-create reuse `workspaces.create` against the selected workspace's project.
  4. **Single source-of-truth hotkey registry shared by the App shell AND TerminalPanel.** `apps/desktop/src/renderer/shortcuts/registry.ts` defines id → default chord → description; chords are encoded from `KeyboardEvent.code` (layout-independent, capture-friendly). The renderer loads overrides on mount and BOTH keymaps read the merged config. App-level chords (`Ctrl+Alt+↑/↓` prev/next, `Ctrl+Alt+N` quick-create, `Ctrl+Alt+Shift+N` new dialog, `Ctrl+Alt+O` open project, `Ctrl+,` settings) were chosen to never collide with TerminalPanel's `Ctrl+Shift+*` and `Ctrl+1–9`. The numeric preset slots (`Ctrl+1–9`) stay a parametric rule, not nine rebindable entries.
  5. **Shortcuts persisted in PGlite via the existing `hotkey_overrides` table (scope `"desktop"`).** New `settings` router: `getHotkeys`/`setHotkey`/`setHotkeys`/`resetHotkey`/`resetHotkeys`. The store upsert is **delete-then-insert** (not `ON CONFLICT`) because Postgres treats `NULL os_scope` as DISTINCT on the `(action_id, os_scope)` unique index, which would otherwise let a NULL-scoped binding duplicate.
  6. **Dialogs are mounted only while active.** The `@swarm/ui` `Dialog` renders a *closed* native `<dialog>` visibly (its `flex` class overrides the UA `dialog:not([open]){display:none}`), so an always-mounted dialog pollutes the page (intercepts clicks, duplicates accessible names). The renderer conditionally mounts New/Open/Settings dialogs.
  7. **E2E open-external test seam (not a mock).** `GROVE_EXTERNAL_LAUNCH_CAPTURE=<file>` makes the host record the launch it WOULD perform (one JSON line) instead of spawning a GUI app, without requiring the binary to exist — so the real journey is asserted on a headless runner. The fixture project gains a REAL `localPath` repo and a temp `worktreesRoot` so New-worktree cuts a genuine git worktree.
- **Consequences:** P08/P09 land REAL + wired to the host. Local gate green: Biome lint clean; `turbo run typecheck build test --force` 17/17 + 26/26; banned-token scan empty; Playwright e2e 8/8 (4 new B2 + 4 prior) against the real seeded host; new host round-trip tests (`settings-projects.test.ts`) 6/6. Per ADR-0012 these are **staged** (no version cut / no 3-OS CI) until billing is restored. _(Resolved: billing unblocked by making the repo public → run 27550630185 green 3 OSes + e2e → shipped in v0.4.0.)_

## ADR-0014 — Phase-4 mobile PWA: host-served + QR-pairing connection; secure-context features validated at localhost, secure LAN/remote origin deferred to Phase-5 (Caddy/tunnel)
- **Date:** 2026-06-15 · **Status:** Accepted
- **Context:** Phase 4 builds `apps/mobile` as an installable, offline-first PWA with full agent orchestration from the phone (P12), on `@swarm/ui` + the REAL host. Two problems have no spec answer and (per §0.2) are resolved here without human input: (1) **a phone browser cannot read `~/.grove/host/manifest.json`** (the desktop's Electron-only endpoint+token source), so the PWA needs another way to obtain the host endpoint + bearer; (2) **service workers + Web Push require a *secure context*** (HTTPS or `localhost`) — a LAN IP over plain `http` is NOT secure, so SW registration + push would silently fail on the bare-LAN path.
- **Decisions:**
  1. **Host serves the PWA (same-origin).** Mount `serveStatic(apps/mobile/dist)` on the existing Hono host server so the host IS the PWA origin — no CORS/origin ambiguity, and on `localhost` it is a secure context.
  2. **QR / single-use-code pairing — the bearer never travels in a QR/URL/SW cache.** Add a `pair` router: the CLI prints a QR encoding `endpoint + a short-lived, single-use code` (NOT the 256-bit bearer); a **public** `pair.redeem({code})` exchanges code → `{ endpoint, token, resumeToken }`; the PWA stores the bearer in **IndexedDB** (never precached by the SW). Preserves private-by-default (P11): the bearer is never exposed in transit artifacts.
  3. **LAN bind is an explicit opt-in.** The host stays bound to `127.0.0.1` by default (P11); a `--lan`/`--host 0.0.0.0` flag is required to expose it — paired with the single-use code, not an open port.
  4. **Secure-context features are validated at `localhost` in Phase 4; the secure LAN/remote origin is Phase 5.** The full PWA (manifest, install, injectManifest SW, Web Push/VAPID) is BUILT in Phase 4 and **validated by the phone-viewport Playwright e2e served at `localhost` (a secure context)** — so install + SW + push are really proven, not faked. On-device phone access over the LAN is HTTP (responsive client + pairing + live tRPC/sync work; SW/push light up only on a secure origin). Delivering that secure origin on the LAN/remotely (Caddy internal TLS + cloudflared/localtunnel) is **Phase 5** — consistent with ADR-0009 (Caddy → Phase 5) — at which point install + push fully light up on the phone. This keeps Phase 4 a real, testable product without pulling the whole TLS/tunnel stack forward or asking the user.
  5. **Web Push/VAPID via `web-push` (MIT).** Implement the ALREADY-DEFINED `@swarm/api` `NotificationsRouter` (`list`/`markRead`/`subscribePush`) against the ALREADY-PRESENT `notifications` + `push_subscriptions` PGlite tables; generate + persist a VAPID keypair; send on `workspace.status_changed → needs_attention`. Document iOS limits (installed-PWA only, 16.4+, user-gesture).
  6. **SW strategy: `vite-plugin-pwa` injectManifest (custom SW).** Precache the app-shell ONLY; network-first for API; **never** cache the bearer / tRPC responses / auth. New `BottomNav` primitive + `Sheet`-driven single-column nav for phone (the desktop shell's no-reflow-below-16rem is unchanged).
- **Build waves:** W1 app skeleton (Vite+React+Tailwind on `@swarm/ui`, manifest, bottom-nav shell) → W2 host static-serve + `pair` router + pairing screen (IndexedDB bearer) → W3 core read journeys (workspace list/switch, live status, diff review) → W4 terminal-over-WS w/ touch accessory bar + dispatch/quick-create → W5 offline SW + `notifications`/VAPID push → W6 phone-viewport e2e + §6.3 evidence + independent Critic → v0.5.0.
- **Consequences:** The phone gets a real, installable, push-capable PWA proven at a secure context, with a private-by-default pairing model and no new paid services. New host surface: `serveStatic` mount, `pair` router, `notifications` router + VAPID send path, LAN-bind flag. Risks tracked: touch-xterm input, SW/token hygiene, iOS push limits, and the LAN secure-context gap closed in Phase 5.
- **W5 implementation notes (2026-06-15) — offline service worker + Web Push/VAPID on `@swarm/ui` + the real host (P12):**
  - **Custom injectManifest SW, app-shell precache only, API never cached.** `vite-plugin-pwa@1.3.0` (MIT, supports Vite 8) compiles `apps/mobile/src/sw.ts` → `dist/sw.js`, injecting the 67-entry app-shell precache manifest (HTML/CSS/JS/icons/`manifest.webmanifest`, `globPatterns` js/css/html/svg/png/woff/woff2/webmanifest). The SW (`workbox-precaching`/`-routing`/`-strategies` v7, all MIT) `precacheAndRoute`s the shell, registers a `NetworkOnly` route for the API/auth surface (`/trpc`, `/sync`, `/terminal`, `/healthz`, `pair.*`, and **any** request bearing an `Authorization` header — the bearer/tRPC responses are NEVER cached), and a `NavigationRoute` offline-shell fallback denylisting those API paths. `/sync` + `/terminal` are WS upgrades the SW does not intercept at all. `skipWaiting` + `clients.claim` so a first visit is SW-controlled at once (the e2e asserts `serviceWorker.controller`). Push handlers in the SW: `push` → `showNotification`, `notificationclick` → focus/navigate.
  - **Registration is manual (not `virtual:pwa-register`).** `vite-plugin-pwa`'s `registerSW` pulls `workbox-window`, which rolldown/Vite 8 could not resolve from the app's isolated `node_modules`; the self-contained SW bundle has no top-level imports, so `main.tsx` registers `/sw.js` directly (`navigator.serviceWorker.register`, classic, on `window load`, failure non-fatal). The hand-authored `public/manifest.webmanifest` is kept (`manifest:false`, `injectRegister:false`). The SW is typechecked by a dedicated `tsconfig.sw.json` (`lib: WebWorker`) chained into the app `typecheck` (`src/sw.ts` excluded from the DOM-lib main config to avoid lib clashes).
  - **Host `notifications` router + VAPID generated-once-and-persisted + send-on-needs_attention.** `apps/host/src/push.ts`: `loadOrCreateVapid(manifestDir)` generates a `web-push` (MIT) VAPID keypair on first run and persists `vapid.json` (reused thereafter — rotating would orphan every device's subscription). `web-push` is CommonJS: under Node ESM its named bindings live on the synthetic `default` (Bun exposes them on the namespace), so the import picks whichever carries the functions — runs under both runtimes (host=Node, unit tests=Bun, ADR-0007a). `HostServices.vapidPublicKey` is added; the `notifications` router (`trpc.ts`) implements the `@swarm/api` contract: `vapidPublicKey` (PWA's app-server key), bearer-gated `subscribePush` (upsert by unique endpoint), `list`/`markRead`. New `@swarm/db` store helpers (`createNotification`/`listNotifications`/`markNotificationRead`/`upsertPushSubscription`/`listPushSubscriptions`/`removePushSubscription`) against the already-present `notifications`+`push_subscriptions` tables. `server.ts` subscribes a listener to the SAME `EventLog` the orchestrator writes: on `workspace.status_changed → needs_attention` it records a notification row and `PushSender` pushes to every stored device; a 404/410 prunes a dead subscription.
  - **Sub-decision — the web-push transport seam is at the NETWORK boundary, never a user-path mock.** Production calls real `webpush.sendNotification` (live HTTPS POST). `GROVE_PUSH_CAPTURE=<file>` (set ONLY by tests) instead runs the REAL VAPID signing + payload encryption via `webpush.generateRequestDetails` and records the request (endpoint + presence of the VAPID `Authorization` JWT + encrypted body) to a JSON-lines file — proving a genuine push was produced without a live push service. The host unit/integration test (`notifications.test.ts` + Node `notifications-worker.ts`) drives subscribe → REAL `needs_attention` event on the shared log → captured web-push to the stored subscription (genuine P-256 ECDH keys, the exact browser shape) → notification row → `markRead`, 7/7.
  - **Sub-decision — PWA push UX in Settings (no new tab).** A `NotificationsCard` (`apps/mobile/src/host/`) renders a real opt-in (`Enable notifications` → user-gesture `Notification.requestPermission` → `pushManager.subscribe({applicationServerKey})` → `notifications.subscribePush`) plus the inbox (`notifications.list` with per-row `markRead`). Honest states: denied/blocked, and an explicit "this browser can't enable push here" when `subscribe` rejects. ## ADR-0015 — Phase-5 one-command bootstrap `grove up` (TS, cross-shell, detached daemon)
- **Date:** 2026-06-15 · **Status:** Accepted
- **Context:** Spec P13 wants Grove standable "without touching your PC". Today `apps/cli` wires only `host`/`pair`/`status`; `host` runs the daemon in the FOREGROUND; `start`/`stop` print "not wired yet". A real bootstrap must do everything in one command, on PowerShell/cmd AND POSIX, with no `.sh` on user paths (ADR-0004).
- **Decision:** Add **`grove up`** (TypeScript via the bundled CLI): verify deps (node 24 / bun / git; cloudflared optional with a graceful note) → init the PGlite store (`openStore`, ADR-0003) → ensure the bearer + VAPID keypair exist (idempotent — both already persist once) → start the daemon **detached/backgrounded** (not foreground) writing the manifest+PID → print the pairing QR. Idempotent (re-running attaches to / reports the running daemon, never double-starts). Wire **`start` (detached) / `stop` (graceful kill by manifest PID then tree-kill) / `status`** as the lifecycle primitives `grove up` builds on. All cross-shell TS; the only OS-specific bit is detached spawn + PID kill (`child_process` detached + the existing `tree-kill`, never a shell script).
- **Consequences:** One command stands up a private, self-hosted Grove with a scannable pair QR. New CLI surface: `grove up`, real `grove start/stop/status`, a daemon-lifecycle module. Risk: detached-process + PID-kill differ on Windows (handled via `child_process` detached + `tree-kill`/`taskkill /T`, proven in the engine — ADR-0007/0011). Unit-test the dep-verify + a dry-run path.
- **W1 implementation (2026-06-15):** Wired real `grove start`/`stop`/`status` in `apps/cli/src/index.ts` against the live host daemon (no mocks on user paths).
  - **`start` detaches under Node, not the parent runtime.** It re-invokes the CLI itself — `node <cli-index.ts> host <args>` — with `spawn(..., { detached: true })` + `child.unref()` and stdio → `<grove-home>/host/daemon.log`, reusing exactly what `grove host`/`runDaemon` runs but pinned to Node (node-pty can't run under Bun — ADR-0007a). The child writes the manifest+PID; the parent polls `/healthz` (+ a fresh-PID manifest) until reachable, then returns. **Idempotent** (a live PID + healthy `/healthz` is reported, never double-started). **Bounded transient retry:** a detached-boot crash (code 1) is re-spawned up to 3× (the ADR-0011 Windows file-lock class — observed flaking under full-parallel-test load); a hang/timeout is surfaced immediately, never retried.
  - **`stop` is graceful-then-forced tree-kill by manifest PID.** SIGTERM to the process group (POSIX `kill(-pid)`, the daemon being a `detached` group leader) / `taskkill /T` (Windows) first; SIGKILL / `taskkill /T /F` if it survives the grace window; confirms the PID is gone, then clears the manifest so `status` reads stopped. A SIGTERM/SIGINT handler on the daemon (`installDaemonShutdown`) runs `host.close()` so the POSIX graceful pass actually tears down sockets+PTYs+manifest; Windows has no deliverable graceful signal for a windowless process, so it falls through to the forced pass (consistent with the engine's existing `killTree`).
  - **`status` checks real liveness, not just the manifest:** manifest present → PID alive (`process.kill(pid,0)`) → `/healthz` answers (bounded fetch). Honest about no-manifest / stale-manifest (dead PID) / process-up-but-silent, and prints uptime from `startedAt`.
  - **Test seam (NOT a mock):** added `groveHome()` to `@swarm/host` — a `GROVE_HOME` env override for the host home (manifest + VAPID), honored by both `defaultManifestPath()` and `startHost`'s `manifestDir` so the daemon and its CLI clients always agree. The mandatory real round-trip (`apps/cli/src/lifecycle.test.ts`) sets `GROVE_HOME` + `--db` to throwaway temp dirs + `--port 0` and drives `start → status → (idempotent) start → stop`, asserting `/healthz` really answers and the PID is really dead — deterministic + bounded, daemon under Node. No `tree-kill` dep was added to `@swarm/cli` (bun.lock unchanged); the established `taskkill /T [/F]` + POSIX-group-signal pattern is reused inline. `grove host` (foreground) is unchanged for debugging. Deferred to W2: `grove up` composes start + pair (+ dep-verify).
- **W2 implementation (2026-06-15):** Added `grove up` — the friendly one-command bootstrap — as a pure COMPOSITION over the W1 lifecycle (no new daemon code): `apps/cli/src/dep-verify.ts` (cross-platform preflight) + `runUp`/`UpResult` + usage text in `apps/cli/src/index.ts`, proven by `apps/cli/src/up.test.ts`. No new dependency (reuses `qrcode-terminal`); **bun.lock unchanged**; the daemon still runs under Node.
  - **Cross-platform dep preflight (ADR-0004 — no shell assumptions).** Each tool is probed by EXECUTING `<bin> --version` via `child_process.execFile` (promisified) and treating `ENOENT`/non-zero-exit/timeout uniformly as "not found" — **no `where`/`which` parsing, no shell**, so it behaves identically on PowerShell/cmd and POSIX (`CreateProcess`/`execvp` resolve the binary off PATH; the required tools are real `.exe`s on Windows, not `.cmd` shims).   - **`grove up` flow + idempotency.** preflight → (`--check`/`--dry-run` stops here, preflight-only) → `runStart` (idempotent; boots the daemon, on which the host creates/loads the PGlite store + bearer + VAPID — so `up` adds NO store/key logic) → mint a single-use code + render the terminal QR via the shared `runPair` (refactored to RETURN `{code,endpoint,url}` so `up` folds it into one summary and tests can assert a real mint) → tidy summary (endpoint, PID, "grove stop to shut down"). `--port`/`--db`/`--host`/`--lan` pass through to the daemon; `--lan` also flows to pairing. **Never throws:** every error is caught into a friendly message + non-zero exit code (set on `process.exitCode` by the dispatcher, which now also has `grove help`/`--help` usage documenting `up`).

- **Date:** 2026-06-15 · **Status:** Accepted (resolves/supersedes ADR-0009; closes ADR-0014 decision 4)
- **Context:** A phone reaches the host over the internet only via a tunnel. ADR-0009 named cloudflared/localtunnel + Caddy. Recon refines: **cloudflared quick tunnels** (`cloudflared tunnel --url http://127.0.0.1:<port>`, Apache-2.0 binary) need no account and terminate HTTPS at the Cloudflare edge with a **publicly-trusted** cert → **Caddy is NOT needed on the remote path** (host stays loopback behind the tunnel). The PWA already overrides its endpoint to `window.location.origin`, so a tunnel origin flows through pairing unchanged. An **HTTPS origin is a secure context**, so the tunnel is where **on-device SW install + Web Push finally light up** — closing the ADR-0014 dec-4 deferral.
- **Decision:** Default remote path = **cloudflared quick tunnel**; **localtunnel (MIT, self-hostable)** as the fully-OSS fallback. **Caddy's only Phase-5 role is LOCAL evidence:** an internal self-signed TLS reverse-proxy in front of the loopback host to PROVE the secure-context path (SW registers + `pushManager.subscribe` over HTTPS) in a Playwright e2e (`ignoreHTTPSErrors`) — never shipped to phones (they use cloudflared's trusted edge cert). Add `grove pair --remote` (+ `grove host --remote`): spawn the tunnel, parse its URL, set pairing **endpoint = tunnel URL**, add an optional Host/Origin allowlist env (defense-in-depth; the bearer is still the gate), print the QR. **CI-vs-local evidence split (§6.2):** CI proves the **secure-context SW+push over local Caddy self-signed TLS** (deterministic) + the `--dir` packaging build; a **real public cloudflared tunnel + phone-scan pairing trace** is local-only evidence (tunnels aren't deterministic in CI).
- **Consequences:** The phone-only remote workflow is real, OSS, and on-device push works — nothing paid. Risks: TryCloudflare quick-tunnels are a Cloudflare-hosted free service (ToS/availability) → localtunnel fallback keeps it fully OSS; self-signed certs are local-proof only, never sent to devices; the bearer never rides the QR (ADR-0014).
- **W3 sub-decisions (landed):**
  - **Tunnel manager** (`apps/cli/src/tunnel.ts`): `startTunnel({ port, provider, command?, timeoutMs? })` spawns the REAL provider binary via `child_process` and resolves on the first public URL parsed from stdout/stderr with ONE combined matcher `https://…(trycloudflare.com|loca.lt)` (accumulating chunks so a split URL line is caught), bounded by a timeout. Returns `{ url, provider, pid, stop() }`; `stop()` **tree-kills** the process reusing the W1 helper — `treeKill`/`pidAlive` were extracted to `apps/cli/src/proc.ts` so lifecycle + tunnel share one cross-platform kill (`taskkill /T /F` on Windows, negative-PID group on POSIX). **cloudflared is the default; localtunnel (`lt`, MIT) is the auto-fallback on ENOENT** (provider pinnable via `--provider`); a provider that is present-but-broken surfaces an honest error rather than silently switching; neither present ⇒ an install-docs error. No heavy dep added — the binary is invoked (bun.lock unchanged).
  - **`grove pair --remote` / `grove up --remote`** (`apps/cli/src/index.ts`): ensure the daemon (idempotent `runStart`) → open the tunnel → pairing **endpoint = tunnel HTTPS URL** → mint a single-use code (`pair.start`) → render a QR encoding **`<tunnelUrl>/?code=<CODE>`** (`buildPairUrl`, exported for unit assertion) + a "scan from your phone, anywhere" hint; the tunnel is held up until Ctrl-C / `grove stop`. The **bearer is never in the QR** (only the tunnel URL + one-time code; the bearer still arrives via `pair.redeem`). The PWA's `window.location.origin` default makes it talk to the tunnel origin unchanged, and that HTTPS origin is the secure context where on-device SW install + Web Push finally light up (**closes ADR-0014 dec-4**).
  - **Host defense-in-depth** (`apps/host/src/origin-allowlist.ts` + `server.ts`): optional **`GROVE_ALLOWED_ORIGINS`** narrows CORS Origin reflection to listed origins when set; **default stays permissive** (the bearer is the real gate, P11) and loopback/RFC-1918 LAN are ALWAYS allowed so same-origin/LAN/loopback never break.
  - **Test seam (no user-path mock):** a committed stub binary (`apps/cli/src/__fixtures__/stub-tunnel.mjs`) is driven via the explicit `command` override — the REAL cloudflared/localtunnel binary runs on every user path. `tunnel.test.ts` asserts URL parse, that `stop()` leaves the PID dead, and that a missing/never-printing binary yields a bounded honest error; the real public tunnel + phone-scan trace remain **W6 local evidence** (tunnels aren't deterministic in CI).
  - **Gate note:** bun's 5 s default *hook* timeout was insufficient for the W1/W2 daemon-teardown `afterAll` under heavy parallel `turbo` load (graceful→forced tree-kill + Windows `rm` retries), so those two hooks got an explicit 30 s timeout (assertions unchanged). `turbo … --force` at default concurrency intermittently flakes on this Windows host (pty-supervisor PTYs + daemon teardown contending); it is deterministically green at `--concurrency=1` (45/45 tasks) and every package passes in isolation — consistent with the project rule "local `turbo --force` ≠ CI".
- **Phase-5 wave plan:** W1 daemon lifecycle (real `start`/`stop`/`status`, detached + PID-kill) → W2 `grove up` one-command bootstrap → W3 remote path (`grove pair --remote` cloudflared/localtunnel + tunnel-URL pairing → on-device push) → W4 electron-builder packaging config → W5 CI evidence (`package` job `--dir` on 3 OS + a Caddy self-signed-TLS secure-context SW/push e2e) → W6 local evidence (installers + Windows GUI launch + remote-pairing trace under `evidence/phase-5/`, closing P14) + independent Phase-5 Critic → v0.6.0 + Linear P13 (GHO-377) & P14 (GHO-378) Done.
